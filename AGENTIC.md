## Overview

The agent is a multi-step, tool-calling financial research assistant built with [LangGraph](https://github.com/langchain-ai/langgraph). Its purpose is to help users analyze public companies through valuation-oriented financial analysis: fetching XBRL filings, market data, sector data, computing growth rates and financial ratios, and synthesizing investor-grade summaries.

The core loop follows a Plan → Execute → React cycle with a hard LLM call budget to prevent runaway execution. Every routing decision is made by an LLM using structured output (Pydantic models via function calling), making the graph deterministic enough to reason about while still being flexible enough to handle a wide range of user requests.

## Agent State

**Defined in:** [backend/src/backend/agent/state.py](backend/src/backend/agent/state.py)

All nodes share a single `AgentState` TypedDict that flows through the graph. LangGraph merges updates from each node into this shared state.

```python
class AgentState(TypedDict):
    messages:        Annotated[list[AnyMessage], operator.add]  # Chat history; new messages are appended
    llm_calls:       int                                         # Running count of LLM invocations
    max_llm_calls:   int                                         # Budget cap (default: 5)
    context:         str                                         # Static app_context string injected into every LLM prompt
    current_year:    int                                         # Calendar year at invocation time
    available_tools: list[dict[str, Any]]                       # Serialized tool definitions for prompts
    tool_plan:       list[dict[str, Any]]                       # Ordered list of planned tool calls
    tool_results:    dict[str, list[dict[str, Any]]]            # Cached results keyed by tool name
    route:           str                                         # Last routing decision; used for debugging
```

### Key state invariants

- `messages` uses `operator.add` as its reducer, meaning every node that appends messages does so additively — no node overwrites the full history.
- `tool_results` is a dict mapping `tool_name → list[{args, result}]` so multiple calls to the same tool (e.g., for two different tickers) are stored as a list.
- `tool_plan` is a list of `{tool_name, args, reason}` dicts generated by the planner or extended by react.

## Graph Architecture

**Defined in:** [backend/src/backend/agent/graph.py](backend/src/backend/agent/graph.py)

The graph is a `StateGraph(AgentState)` compiled by LangGraph. All routing is done via `Command(goto=..., update=...)` returns from node functions — there are no `add_conditional_edges` calls. This means routing logic lives entirely inside the node functions themselves.

### Node registry

| Node constant        | Function           | Role |
|----------------------|--------------------|------|
| `router`             | `router()`         | Entry gate; decides if tools are needed |
| `plan_node`          | `plan_node()`      | Creates the ordered tool execution plan |
| `market_data_node`   | `market_data_node()` | Executes market data tools |
| `sector_data_node`   | `sector_data_node()` | Executes sector data tools |
| `financials_node`    | `financials_node()` | Executes financial statement tools |
| `growth_rates_node`  | `growth_rates_node()` | Executes growth rate tools (depends on financials) |
| `ratios_node`        | `ratios_node()`    | Executes ratio tools (depends on financials) |
| `react_node`         | `react_node()`     | Reviews gathered data; answers or requests more |

The only static edge is `START → router`. Every other transition is decided dynamically by the node.

### Graph diagram

A PDF rendering of the Mermaid-based topology diagram is saved automatically to [backend/src/backend/agent/agent_graph.pdf](backend/src/backend/agent/agent_graph.pdf) each time `initialize_agent()` is called.

### Route aliases

The LLM returns route strings that do not always match node names exactly. `ROUTE_ALIASES` normalizes them:

```python
ROUTE_ALIASES = {
    "plan":         "plan_node",
    "market_data":  "market_data_node",
    "sector_data":  "sector_data_node",
    "financials":   "financials_node",
    "growth_rates": "growth_rates_node",
    "ratios":       "ratios_node",
    "react_node":   "react_node",
    "end":          "end",
}
```

## Nodes in Detail

### `router`

**Purpose:** Determines whether the user's request needs tool execution or can be answered immediately.

**LLM schema:**
```python
class RouterDecision(BaseModel):
    route: Literal["plan_node", "end"]
    answer: str = ""   # Non-empty only when route == "end"
```

**Routing logic:**
- `end` → appends an `AIMessage` with the answer (or a default prompt), goes to `END`
- `plan_node` → increments `llm_calls`, proceeds to planning

**Fallback:** Any exception routes to `END` with an error message.

---

### `plan_node`

**Purpose:** Translates the user's request into a structured, ordered list of tool calls.

**LLM schema:**
```python
class PlanDecision(BaseModel):
    route: Literal["market_data_node", "sector_data_node", "financials_node", "end"]
    tool_plan: list[ToolStep] = []
    answer: str = ""   # Non-empty only when route == "end"
```

```python
class ToolStep(BaseModel):
    tool_name: str
    args: dict[str, Any] = {}
    reason: str = ""
```

**Routing logic:**
- The LLM picks the first node to activate based on which tool category the first planned step belongs to.
- `financials_node` is also the correct route when the plan includes growth or ratio tools (since those depend on financials being fetched first).
- `end` is returned with an `answer` if required information is missing.

**Debug output:** The full plan is printed to stdout as JSON.

---

### `market_data_node` / `sector_data_node`

**Purpose:** Execute tools in their respective groups directly from the planner-provided arguments.

Both delegate to `_data_tool_node()`, which calls `run_planned_tools()` and always routes to `react_node` on completion.

---

### `financials_node`

**Purpose:** Fetches financial statement data (income statement, balance sheet, cash flow). This node must always run before `growth_rates_node` or `ratios_node` because those tools inject the `HistoricalFinancials` object as input.

**Routing after execution:** `next_financial_dependent_route()` is called — if any growth or ratio steps remain in the plan, the node routes to them; otherwise it goes to `react_node`.

---

### `growth_rates_node` / `ratios_node`

**Purpose:** Execute growth-rate and ratio tools by injecting the `HistoricalFinancials` result gathered by `financials_node`. The LLM plan stores only `{ticker: "AAPL"}` as args; the actual `HistoricalFinancials` object is looked up from `tool_results` and injected at runtime by `run_financial_dependent_tools()`.

Both always route to `react_node` on completion.

---

### `react_node`

**Purpose:** The synthesis and continuation-decision node. After data nodes finish, react reviews all gathered results and either composes a final answer or requests additional tool calls.

**Pre-check:** Before invoking the LLM, react calls `next_route_from_tool_plan()`. If there are still unfinished planned steps, react dispatches directly to the responsible node without spending an LLM call.

**Budget enforcement:**
- If `llm_calls >= max_llm_calls`: returns `fallback_react_answer()` immediately, no LLM call.
- If `llm_calls == max_llm_calls - 1` (last allowed call): `force_final_answer=True` is passed in context, instructing the LLM to answer rather than plan more work.

**LLM schema:**
```python
class ReactDecision(BaseModel):
    route: Literal[
        "market_data_node", "sector_data_node", "financials_node",
        "growth_rates_node", "ratios_node", "end"
    ]
    tool_plan: list[ToolStep] = []
    answer: str = ""   # Non-empty only when route == "end"
```

**Tool plan extension:** When react requests more work, `extend_tool_plan()` merges the new steps into the existing plan without duplicating already-completed steps.

## Execution Flow

```
User input
    │
    ▼
START ──► router
            │
     ┌──────┴──────┐
   "end"        "plan_node"
     │               │
     ▼               ▼
   END           plan_node
                     │
          ┌──────────┼──────────┐
          ▼          ▼          ▼
  market_data_  sector_data_ financials_
     node          node         node
          │          │          │
          │          │    ┌─────┴──────────┐
          │          │    ▼                ▼
          │          │ growth_rates_   ratios_
          │          │    node          node
          │          │    │                │
          └──────────┴────┴────────────────┘
                          │
                          ▼
                      react_node
                          │
               ┌──────────┴──────────┐
               ▼                     ▼
        (more tools)              "end"
        (loop back)                  │
                                     ▼
                                    END
```

### Typical request lifecycle (e.g., "Analyze Apple's profitability")

1. **router** — routes to `plan_node` (in-scope, tool-requiring request)
2. **plan_node** — creates plan: `[get_financials(AAPL), get_profitability_ratios(AAPL)]`; routes to `financials_node`
3. **financials_node** — calls `get_financials("AAPL")`; detects pending ratio steps; routes to `ratios_node`
4. **ratios_node** — calls `get_profitability_ratios(hf=<HistoricalFinancials>)`; routes to `react_node`
5. **react_node** — all planned steps complete; LLM composes final investor analysis; routes to `END`

## Tool System

### Tool metadata decorator

**Defined in:** [backend/src/backend/services/tool_metadata.py](backend/src/backend/services/tool_metadata.py)

Each tool is decorated with `@agent_tool(...)` which attaches a `metadata.agent` dict:

```python
{
    "group":               "financial_statement",  # Tool category
    "route":               "financials",           # Graph node that executes it
    "capability":          "...",                  # Human-readable description for prompts
    "requires_financials": False,                  # True for growth/ratio tools
}
```

### Available tools

| Tool | Group | Route | Requires Financials |
|------|-------|-------|---------------------|
| `get_financials` | `financial_statement` | `financials` | No |
| `get_market_data` | `market_data` | `market_data` | No |
| `get_sector_data` | `sector_data` | `sector_data` | No |
| `get_income_statement_growth_rates` | `growth_rate` | `growth_rates` | Yes |
| `get_balance_sheet_growth_rates` | `growth_rate` | `growth_rates` | Yes |
| `get_liquidity_ratios` | `ratio` | `ratios` | Yes |
| `get_solvency_ratios` | `ratio` | `ratios` | Yes |
| `get_profitability_ratios` | `ratio` | `ratios` | Yes |

### Tool data sources

- **`get_financials`** — XBRL filings from SEC Edgar (5–10 years of income statement, balance sheet, cash flow, per-share data)
- **`get_market_data`** — current price, beta, shares outstanding, market cap from Yahoo Finance; optional risk-free rate from FRED DGS10
- **`get_sector_data`** — equity risk premium by year from Damodaran's published data

### Tool execution paths

**Direct tools** (`run_planned_tools`): `get_financials`, `get_market_data`, `get_sector_data` — called with args directly from the plan.

**Dependent tools** (`run_financial_dependent_tools`): `get_*_growth_rates`, `get_*_ratios` — the plan stores only `{ticker: "AAPL"}` for these; at runtime, the matching `HistoricalFinancials` result is looked up from `tool_results["get_financials"]` and injected as the actual function argument. This is done in `_financial_dependent_args()`:

```python
# Checks whether the tool expects "hf" or "financials" as the parameter name
if "hf" in tool_args:
    return {"hf": financials}
if "financials" in tool_args:
    return {"financials": financials}
```

### Tool result caching

Results are stored in `state["tool_results"]` as `{tool_name: [{args, result}, ...]}`. Before executing any planned step, `planned_step_done()` checks whether a result with matching normalized args already exists:

- For growth/ratio tools: matches on ticker only (the `HistoricalFinancials` object is ignored for comparison)
- For other tools: matches on all args after stripping nested dict refs

## LLM Configuration

**Defined in:** [backend/src/backend/core/llm.py](backend/src/backend/core/llm.py)

The LLM is configured via environment variables loaded from a `.env` file:

```python
LLM_PROVIDER = os.getenv("LLM_PROVIDER")   # e.g., "openai", "anthropic"
LLM_MODEL    = os.getenv("LLM_MODEL")       # e.g., "gpt-4o", "claude-3-5-sonnet-20241022"

CHAT_MODEL = init_chat_model(
    model=LLM_MODEL,
    model_provider=LLM_PROVIDER,
    temperature=0,                            # Deterministic output
)
```

`CHAT_MODEL` is a LangChain `BaseChatModel`. It is imported into `state.py` and used in `graph.py` via `CHAT_MODEL.with_structured_output(schema, method="function_calling").invoke(messages)`.

Setting `temperature=0` ensures routing decisions are reproducible for the same input.

### How structured output works

Every LLM call in the agent uses `.with_structured_output(PydanticModel, method="function_calling")`. This forces the LLM to respond with a JSON object that matches the schema, which is then automatically parsed into a Pydantic model instance. If parsing fails, the node catches the exception and routes to `END` with an error message.

## Prompts

**Defined in:** [backend/src/backend/agent/prompts.py](backend/src/backend/agent/prompts.py)

### `app_context`

Injected into every LLM call as `state["context"]`. Establishes the agent's identity, domain, and operating rules:

- Financial research assistant specialized in company valuation
- Must only use tools available in the state
- Must not invent financial data, companies, tickers, or sources
- Must separate facts from assumptions
- Must keep responses concise and investor-oriented

### `router_prompt`

Instructions for the router node. Key behavioral rules:
- Routes to `end` for: vague input, out-of-scope requests, missing required info, educational questions answerable without tools, requests for legal/tax/investment advice
- Routes to `plan_node` for: in-scope requests that identify a company and require data retrieval
- For short confirmations ("yes", "go ahead"), uses prior conversation as context
- Does not ask to confirm well-known tickers unless genuinely ambiguous

### `plan_prompt`

Instructions for the plan node. Key behavioral rules:
- Uses tool descriptions and argument schemas to select tools — not keyword matching
- Includes prerequisite tools explicitly (financials before ratios/growth)
- Handles multi-company and multi-analysis requests with complete coverage
- Infers tickers from well-known company names without asking
- Picks the smallest complete plan that can answer the request
- Routes `financials_node` when the plan includes any growth/ratio tools

### `react_prompt`

Instructions for the react node. Key behavioral rules:
- Reviews gathered data against the user's original question
- Answers if sufficient; requests more tools if materially beneficial and budget permits
- When requesting ratio/growth tools for already-fetched companies, uses only `{ticker: ...}` in args — does not copy `HistoricalFinancials` objects into the plan
- Obeys `force_final_answer` when set; never asks permission before requesting more data
- Must not repeat already-completed tool calls

### Runtime context injection

Every LLM call in `_invoke_structured_llm()` injects a JSON `Runtime context` block alongside the prompt:

```json
{
  "request_context": "<app_context string>",
  "latest_user_message": "<most recent HumanMessage content>",
  "current_year": 2026,
  "available_tools": [...],
  "tool_groups": {
    "financial_statement_tools": [...],
    "market_data_tools": [...],
    "sector_data_tools": [...],
    "growth_rate_tools": [...],
    "ratio_tools": [...]
  },
  "route_options": ["...", "..."],
  // react-only:
  "remaining_llm_calls_after_this_node": 2,
  "force_final_answer": false,
  "current_tool_plan": [...],
  "gathered_tool_results": {...}
}
```

This context is serialized as JSON and appended to the prompt string before invoking the LLM.

## Helper Utilities

**Defined in:** [backend/src/backend/agent/graph_helpers.py](backend/src/backend/agent/graph_helpers.py)

### Tool group sets

Built at module load time by scanning tool metadata:

```python
FINANCIAL_STATEMENT_TOOLS = _tools_in_group("financial_statement")
MARKET_DATA_TOOLS         = _tools_in_group("market_data")
SECTOR_DATA_TOOLS         = _tools_in_group("sector_data")
GROWTH_TOOLS              = _tools_in_group("growth_rate")
RATIO_TOOLS               = _tools_in_group("ratio")
```

### Routing helpers

- **`next_route_from_tool_plan(state, tool_results)`** — scans the tool plan for the first unfinished step and returns its node route. Returns `"react_node"` when all steps are complete.
- **`next_financial_dependent_route(state, tool_results)`** — after financials are fetched, finds the first pending growth/ratio step. Returns `"react_node"` if none remain.
- **`route_for_tool(tool_name)`** — looks up the routing metadata for any tool by name.

### Tool execution

- **`run_planned_tools(state, tool_names)`** — iterates the tool plan, skips already-done steps, calls `tools_by_name[tool_name].invoke(args)`, accumulates results and `ToolMessage` objects.
- **`run_financial_dependent_tools(state, tool_names)`** — same iteration, but injects matching `HistoricalFinancials` from `tool_results["get_financials"]` into the tool args. If no matching financials exist, produces an error `ToolMessage`.

### Deduplication

- **`planned_step_done(step, tool_results)`** — checks if a result for this step already exists. For growth/ratio tools, matches on ticker. For others, normalizes args (removes nested `$ref` objects) and compares.
- **`extend_tool_plan(current_plan, new_steps)`** — appends new steps only if they are not already in the plan (checked via `_plan_contains_step`).

### Serialization

- **`serialize_tools()`** — converts all registered LangChain tools to JSON-friendly dicts for prompt injection.
- **`serialize_tool_groups()`** — groups tools by category with `{name, capability, requires_financials}` entries.
- **`tool_results_for_prompt(tool_results)`** — converts Pydantic model results to plain dicts using `model_dump()` for safe JSON serialization.
- **`messages_for_llm(messages)`** — filters out `ToolMessage` instances before sending history to the LLM (required for OpenAI chat completions, which reject orphan tool messages without a corresponding tool call reference).

### Fallback

- **`fallback_react_answer(state)`** — produces a plain-text explanation when the LLM call budget is exhausted, listing any tools that were successfully called.

## Data Models

**Defined in:** [backend/src/backend/processing/schema.py](backend/src/backend/processing/schema.py)

These Pydantic models represent the structured data returned by tools:

| Model | Description |
|-------|-------------|
| `CompanyMetadata` | CIK, name, SIC code, industry, fiscal year end, state, website, phone |
| `IncomeStatement` | Revenue, COGS, gross profit, EBIT, net income, interest expense, depreciation |
| `BalanceSheet` | Current assets, cash, total assets, inventory, long-term debt, liabilities, equity; computed: `net_working_capital` |
| `CashFlowStatement` | Net income, capex, D&A, CFO; computed: `fcf = cfo - capex` |
| `PerShare` | Basic and diluted share counts |
| `FinancialPeriod` | Groups one period's income statement, balance sheet, cash flow, and per-share data |
| `HistoricalFinancials` | Ticker, metadata, list of `FinancialPeriod`; has `to_dataframe()` method |
| `MarketData` | Current price, beta, shares outstanding, market cap, optional risk-free rate |
| `SectorData` | Equity risk premium, long-term growth rate (default 2.5%) |

`HistoricalFinancials` is the central object in the dependency chain: `get_financials` produces it; `get_*_growth_rates` and `get_*_ratios` consume it.

```python
    "llm_calls":       0,
    "max_llm_calls":   5,

}
```

## Design Patterns

### Structured output for routing

Every routing decision uses `CHAT_MODEL.with_structured_output(Schema, method="function_calling")`. This gives the LLM a strongly typed interface — it cannot return ambiguous route strings or skip required fields. Pydantic validation happens automatically; exceptions are caught per-node and result in a graceful `END` with an error message.

### Command-based transitions

All node functions return `Command(goto=next_node, update={...})` rather than plain dicts. This is LangGraph's explicit routing mechanism — the node declares its successor directly, so there is no global routing table to maintain.

### LLM call budget

`max_llm_calls` (default 5) caps the total number of LLM invocations per user request. The counter increments in every node that calls `_invoke_structured_llm()`. React checks the budget before invoking the LLM and passes `force_final_answer=True` on the last allowed call, preventing the agent from entering an infinite planning loop.

### Dependency injection for financial tools

Growth and ratio tools require a `HistoricalFinancials` object, but passing large objects through the LLM's tool plan is fragile and wasteful. Instead:
- The LLM plan records only `{ticker: "AAPL"}` for these steps.
- At execution time, `run_financial_dependent_tools()` retrieves the matching `HistoricalFinancials` from `tool_results` and injects it into the function call.
- This means the LLM never needs to see or reproduce the full financial object.

### Tool plan deduplication

`extend_tool_plan()` and `planned_step_done()` together prevent the agent from calling the same tool twice for the same ticker. Normalization strips nested financial objects from comparison so that two steps that differ only in their injected payload are still recognized as duplicates.

### ToolMessage filtering

OpenAI's chat completions API rejects conversation histories that contain `ToolMessage` entries without a preceding `tool_calls` block in an `AIMessage`. Since the agent rebuilds its message history across turns and nodes produce `ToolMessage` objects that are not paired with standard `AIMessage.tool_calls`, `messages_for_llm()` strips all `ToolMessage` instances before every LLM call.

### Graph diagram auto-export

`initialize_agent()` always exports the compiled graph topology to `agent_graph.pdf` using LangGraph's built-in Mermaid renderer. This means the PDF is always in sync with the current code.
